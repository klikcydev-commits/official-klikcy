# Project README

**Klikcy** — production marketing website for a **full-service U.S. digital agency** (web development, app & software, AI automation, e-commerce, branding, growth, and search visibility). SEO/AEO/GEO is one practice area — not the whole company identity.

> **Important:** This repository is a **Vite + React + TypeScript** single-page application (SPA). It is **not** a Next.js project. There are no App Router routes, server components, or `getServerSideProps`. Pages are client-rendered React routes; production output is static files in `dist/`.

---

## 1. Website Overview

### What this site is

Klikcy.com is the public marketing website for **Klikcy** — a remote-first **digital agency** focused on building and growing digital products, not an SEO-only shop.

**Primary positioning (strongest focus):**

1. **Web development** — custom websites, WordPress, Shopify, Next.js/React, landing pages, redesigns, performance  
2. **App & software development** — web apps, SaaS, mobile apps, PWAs, dashboards, APIs, databases  
3. **AI automation** — agents, chatbots, workflows, CRM automation  
4. **E-commerce** — Shopify, WooCommerce, checkout and catalog optimization  
5. **Branding & UI/UX** — identity, design systems, conversion-led design  
6. **Marketing & growth** — CRO, analytics, content systems  
7. **SEO, AEO & GEO** — search and answer-engine visibility as a **supporting** discipline across builds  

**Also covered:** technical hosting, DNS, email, migrations, security, monitoring.

### What Klikcy is not

- Not positioned as a “ranking agency” or SEO-only vendor  
- Not claiming guaranteed #1 rankings — copy uses *built to compete*, *structured for visibility*, *optimized for discoverability*

### Business goals

1. **Convert** leads for web, app, AI, and e-commerce projects (not only SEO retainers).  
2. **Scale** service and local-intent discovery (65 services × 52 states/DC × cities).  
3. **Present** a premium technical brand (design, motion, accessibility).  
4. **Support** Google and AI crawlers with schema, FAQs, and clear service semantics.

### Content model

No CMS. Copy lives in `src/lib/*`, `src/content/*`. Metadata is generated by **service category** in `src/lib/metadata/` (see [Section 6](#6-metadata-system)).

| State hub | `/service-areas/:slug` | `StatePage.tsx` + `src/lib/states.ts` | SPA; 52 states + DC |

| Service × city | `/:service/:state/:city` | `ServiceCityPage.tsx` | **Programmatic** SPA |
| 404 | unmatched | `NotFound.tsx` | SPA |

### Static vs dynamic vs programmatic

| Classification | Meaning in this project |
|----------------|-------------------------|
| **Static routes** | Fixed paths declared in `App.tsx` (`/`, `/about`, `/contact`, `/all-services`, `/services/:slug`, etc.) |
| **Dynamic params** | Slugs resolved at runtime from `services`, `categories`, `states`, `cities` |
| **Programmatic pages** | `ServiceStatePage` / `ServiceCityPage` combine service catalog + geography data + `src/lib/content.ts` templates to generate unique sections and FAQs per URL |
| **Pre-rendered HTML per URL** | **Not implemented** — each URL shares one `index.html`; crawlers that do not execute JavaScript only see the shell unless prerendering is added at the host |

### Route ordering (critical)

In `App.tsx`, **specific routes are registered before** catch-all patterns:

```text
/service-areas/...  →  /services/:slug  →  /:service/:state  →  /:service/:state/:city
```

This prevents `/about` or `/contact` from being captured by `/:service/:state`.

### Legacy URL redirects

`src/lib/legacy-service-slugs.ts` maps old service slugs (e.g. `web-development`) to current catalog slugs. `ServicePage.tsx` redirects with `Navigate` before rendering.

## 5. SEO Strategy

Klikcy’s SEO strategy supports a **full digital agency** catalog. Search visibility is essential, but metadata and copy lead with **web development, app development, software, AI, and e-commerce** — SEO/AEO/GEO appears where it belongs (category `seo-aeo` and as a layer on web builds).

### Priority order (messaging)

1. Web development & custom websites  
2. App, SaaS, and software systems  
3. AI automation & agents  
4. E-commerce  
5. Branding / UI/UX & growth  
6. SEO, AEO, GEO (dedicated services + technical foundation on sites)

### Service-based discoverability

- **65 services** in `src/lib/services.ts` — national URLs `/services/{slug}`.  
- Metadata enhanced at runtime: `buildServicePageMetadata()` in `src/lib/metadata/page-metadata.ts`.  
- **Legacy enrichment** for eight migrated services: `src/content/legacy-service-enrichment.json` + `apply-legacy-enrichment.ts`.

### Web & app page focus

| Vertical | Metadata emphasis |
|----------|-------------------|
| Web | Custom websites, WordPress, Shopify, React/Next.js, conversion, performance, UI/UX; technical SEO as support |
| App / software | Web apps, mobile, SaaS, dashboards, APIs, databases, scalable systems |
| AI | Agents, chatbots, workflows, CRM automation, business process automation |
| SEO/AEO/GEO | Search visibility, answer engines, schema, crawlability, local SEO — **category-specific** |

### Local & programmatic SEO

- **Service areas:** `/service-areas/{state}`, `/service-areas/{state}/{city}` — geographic hubs.  
- **Service × state:** `/{service}/{state}` — e.g. `Web Development in California | Klikcy`, `App Development Company in Texas | Klikcy` (templates vary by vertical + deterministic hash).  
- **Service × city:** `/{service}/{state}/{city}` — e.g. `Web Development in Los Angeles, CA | Klikcy`.  
- Body: `buildServiceStateContent()` in `src/lib/content.ts`; metadata: `buildServiceStateMetadata()` / `buildServiceCityMetadata()`.

### 20 keywords per page (structure)

Generated in `buildKeywords20()` — **not SEO-only**:

| Slot | Count | Examples |
|------|-------|----------|
| Primary | 1 | `{focusKeyword} in {city/state}` or national focus keyword |
| Commercial | 4 | Service-specific stems (web app, SaaS, Shopify, etc.) |
| Location | 5 | `{city} digital agency`, `{state} web development company`, … |
| AEO / GEO / search-intent | 5 | Question-style phrases from vertical profile |
| Semantic | 5 | UX, performance, APIs, automation, etc. |

### AEO and GEO

- Vertical-specific question stems in `src/lib/metadata/positioning.ts` (`aeoQuestionStems`).  
- `buildGeoAeoFaqs()` adds location-aware Q&A on state/city pages; merged into **FAQPage** schema.  
- Dedicated services: `answer-engine-optimization`, `generative-engine-optimization`.  
- Language: *structured for Google and AI search*, *optimized for discoverability* — not “#1 ranking guaranteed.”

### Internal linking, schema, canonicals

- Nav/footer: categories, services, service areas, related services, metro links.  
- Schema: Organization, WebSite, Service, FAQPage, BreadcrumbList (`src/lib/schema.ts`).  
- Canonicals: `https://klikcy.com`; legacy service slug redirects in `legacy-service-slugs.ts`.  
- Sitemaps: ~18k URLs across web, app, AI, and all geo combinations ([Section 7](#7-sitemap-system)).

### Thin / duplicate content

- Titles and descriptions vary by **service + state/city** via `stableHash()` seeds (deterministic, not random).  
- Monitor GSC if programmatic URLs outpace indexation capacity.

## 7. Sitemap System

### Sitemap index

| Sitemap URL | Purpose | Generated from | Notes |
|-------------|---------|----------------|-------|
| `https://klikcy.com/sitemap.xml` | **Sitemap index** — entry point for crawlers | `scripts/split-sitemaps.mjs` | Lists 7 child sitemaps |
| `https://klikcy.com/sitemap-static.xml` | Core marketing pages | Split script bucket `static` | **4 URLs** (`/`, `/about`, `/contact`, `/service-areas`) |
| `https://klikcy.com/sitemap-services.xml` | Services + categories + all-services | Bucket `services` | **74 URLs** (8 categories + 65 services + `/all-services`) |
| `https://klikcy.com/sitemap-areas.xml` | State and city **service-area** hubs | Bucket `areas` | **276 URLs** (`/service-areas/{state}`, `/service-areas/{state}/{city}`) |
| `https://klikcy.com/sitemap-service-state.xml` | **Service × state** programmatic pages | Bucket `serviceState` | **3,315 URLs** (`/{service}/{state}`) |
| `https://klikcy.com/sitemap-service-state-city-1.xml` | Service × city (chunk 1/3) | Bucket `serviceStateCity` | **4,875 URLs** |
| `https://klikcy.com/sitemap-service-state-city-2.xml` | Service × city (chunk 2/3) | Bucket `serviceStateCity` | **4,875 URLs** |
| `https://klikcy.com/sitemap-service-state-city-3.xml` | Service × city (chunk 3/3) | Bucket `serviceStateCity` | **4,875 URLs** |

**Approximate total indexed URLs:** ~18,300 (verify after regeneration).

**Not present in this repo:** blog sitemap, image sitemap, dedicated categories-only sitemap (categories live in `sitemap-services.xml`).

### How URLs enter the sitemap

`npm run sitemap:split` (`scripts/split-sitemaps.mjs`):

1. Reads `public/sitemap.xml` (or child sitemaps if already an index).
2. Collects all `<loc>` page URLs.
3. Ensures core paths exist: `/`, `/about`, `/contact`, `/service-areas`, `/all-services`.
4. **Classifies** each URL by pathname into buckets (static, services, areas, serviceState, serviceStateCity).
5. Writes split `urlset` files and rebuilds the index.

**Generate from source:** `npm run sitemap:generate` (`scripts/generate-sitemaps.mjs`) builds URLs from `states.ts` + `services.ts`, then run `npm run sitemap:split` to chunk files under 50k URLs.

The split script **reorganizes** existing URL lists and synthetic `lastmod` values; run **generate** first when cities or services change.

### `lastmod` handling

- Per-URL `lastmod` is **synthetically varied** by bucket type + URL hash (not tied to Git or file mtimes).
- Sitemap index child `lastmod` uses the latest entry date per file with small jitter.
- Purpose: avoid every URL sharing identical dates; **not** a substitute for real content-update signals.

### Classification rules (pathname)

| Pattern | Bucket |
|---------|--------|
| `/`, `/about`, `/contact`, `/service-areas` | `static` |
| `/all-services`, `/services/*`, `/categories/*` | `services` |
| `/service-areas/{state}`, `/service-areas/{state}/{city}` | `areas` |
| `/{service}/{state}` (two segments) | `serviceState` |
| `/{service}/{state}/{city}` (three segments) | `serviceStateCity` |
| Unclassified | Warning → appended to `sitemap-services.xml` |

### Add or remove pages from sitemaps

1. Add/remove URLs in the underlying urlset data (or generate a new master URL list).
2. Run `npm run sitemap:split`.
3. Confirm counts in script console output.
4. Deploy updated `public/sitemap*.xml` files.
5. Resubmit sitemap index in Google Search Console.

Removing a URL from sitemap **does not** remove it from the site — use redirects or delete the route.

### Test sitemaps locally

```bash
npm run dev
# or after build:
npm run preview
```

Open:

- `http://localhost:8080/sitemap.xml` (dev) or your preview host + `/sitemap.xml`
- Child files: `/sitemap-services.xml`, etc.

Validate XML loads and URLs match live routes (`npm run build` + manual spot check).

### Submit to Google Search Console

1. Verify property for `https://klikcy.com`.
2. Submit **`https://klikcy.com/sitemap.xml`** (index, not individual children).
3. Monitor coverage, excluded pages, and canonical status weekly after large sitemap changes.

```text
Sitemap: https://klikcy.com/sitemap.xml
```

| Topic | Current setup |
|-------|----------------|
| Allowed crawlers | All (`User-agent: *`) |
| Blocked paths | None |
| Sitemap reference | `https://klikcy.com/sitemap.xml` |
| AI crawlers (GPTBot, ClaudeBot, etc.) | **No specific rules** — same as `*` |
| Googlebot / Bingbot | Allowed |

### Safe updates

- Edit `public/robots.txt` directly; redeploy `public/` with `dist/`.
- Use `Disallow` only for intentional exclusions (staging, admin, query params if ever added).
- Keep the `Sitemap:` line pointing at the **index** URL.
- Do not block `/services/` or programmatic paths unless you intend to de-index them.

**Delivery:** `SEO` component → JSON-LD in `<head>`

3. Schema Markup Validator — optional cross-check.

Because JSON-LD is injected client-side, validators must **render JavaScript** (use live deployed URL, not raw `view-source:` only).

## 11. SEO Maintenance Guide

### Add a new service

1. Add `make({ slug, name, category, ... })` in `src/lib/services.ts` (valid `CategorySlug`).
2. Set `related` to existing slugs only.
3. Add links from category page (automatic via `getServicesByCategory`).
4. Add URL to sitemap source list → `npm run sitemap:split`.
5. Deploy; request indexing for `/services/{slug}` in GSC.

### Add a new state or city

1. Edit `src/lib/states.ts` (cities array, `blurb`, `priority`).
2. City slugs auto-derived via `citySlug()` in `src/lib/cities.ts`.
3. Add corresponding `/service-areas/...` and programmatic URLs to sitemap source → split script.
4. Spot-check `ServiceStatePage` / `ServiceCityPage` output for the new geo.

### Update metadata

- Service: `services.ts` (+ legacy JSON if mapped).  
- Homepage: `home.ts` / `Index.tsx`.  
- About: `about.ts`.  
- Global URL: `schema.ts` + sitemaps + robots.

### Update FAQs

- Edit `faqs` arrays in `services.ts` or page content modules.  
- FAQ schema regenerates automatically from the same arrays.

### Check sitemap output

```bash
npm run sitemap:split
```

Verify per-file URL counts in terminal output.

### Check robots.txt

Open `/robots.txt` on staging/production after deploy.

### Test page speed

- `npm run build && npm run preview`  
- Lighthouse (mobile + desktop) on `/`, `/services/{slug}`, programmatic sample URL.

### Broken links

- Crawl with Screaming Frog, Sitebulb, or GSC links report.  
- Pay attention to legacy redirects in `legacy-service-slugs.ts`.

### Verify canonicals

- View rendered `<link rel="canonical">` per template.  
- Ensure HTTP → HTTPS and www → non-www (or chosen host) at CDN/host level.

### Validate schema

- Rich Results Test on representative URLs per template type.

## 12. Deployment Notes

### Build

```bash
npm install
npm run build
```

Output: **`dist/`** (static assets + `index.html`).

### Environment variables

Copy `.env.example` to `.env` (gitignored) and fill in values. **Never commit `.env`.**

| Variable | Purpose |
|----------|---------|
| `SITE_URL` / `VITE_SITE_URL` | Canonical base URL (e.g. `https://www.klikcy.com`) |
| `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS` | Hostinger SMTP for contact form |
| `CONTACT_EMAIL` | Inbox that receives submissions |
| `VITE_CONTACT_EMAIL` | Optional — email shown on the contact page |
| `CONTACT_API_PORT` | Local API port (default `8787`) |

**Local dev with working contact form:**

```bash
npm run dev:all
```

Runs the Vite app and `server/index.mjs` together; Vite proxies `/api/*` to the contact API.

**Production:** static `dist/` alone cannot send mail. Run the Node contact API (`npm run server`) on your host and reverse-proxy `/api` to it, or deploy API + static site on a platform that supports both.

After deploying the contact API, verify it is the latest build:

```bash
curl https://www.klikcy.com/api/health
```

Expect `"version": 2` and `requiredFields` including `"phone"`. If `version` is missing, the live server is still an old build (emails will omit phone).

Contact emails list **all** fields: Name, Email, Phone, Company, Service, Message (plain text + HTML table).

### Hosting requirements

| Requirement | Reason |
|-------------|--------|
| SPA fallback to `/index.html` | Client-side routing |
| Serve `public/` assets at root | `robots.txt`, sitemaps, images copied to `dist/` |
| HTTPS | Canonicals and SEO assume HTTPS |
| `/api` → contact API (production) | Contact form POSTs to `/api/contact` |

### Vercel / Netlify / Cloudflare Pages

Not configured in-repo. Typical setup:

- Build command: `npm run build`  
- Output directory: `dist`  
- Rewrite all routes → `/index.html` (200)

### Post-deploy SEO checks

- [ ] `https://www.klikcy.com/robots.txt`  
- [ ] `https://www.klikcy.com/sitemap.xml` and child sitemaps return 200  
- [ ] Canonical on homepage and sample service page  
- [ ] JSON-LD visible in rendered DOM  
- [ ] Legacy redirects (e.g. `/services/web-development`)  
- [ ] GSC sitemap “Success”  
- [ ] No accidental indexing of staging domain (use `noindex` on staging — **recommended**, not implemented)
