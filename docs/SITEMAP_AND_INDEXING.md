However, there is an architectural risk because two sitemap systems exist.

## Potential “No Referring Sitemaps Detected” Risk
Yes, this is a plausible operational risk.

Why it can happen:
- Search Console may discover URLs independently rather than through the sitemap
- the live endpoint might not be serving the same sitemap file the team expects
- static `public/sitemap.xml` and dynamic `src/app/sitemap.ts` may create ambiguity about which system is authoritative
- if robots and deployment behavior do not consistently expose one stable sitemap source, reporting can become inconsistent

## How Sitemap Should Be Fixed Later
Recommended later cleanup:
1. Choose one canonical sitemap system as the production source of truth.
2. Ensure `robots.txt` references only that sitemap endpoint.
3. Confirm the live endpoint returns the intended XML with `200` status and no redirect.
4. Keep sitemap URLs exactly equal to canonical URLs.
5. Keep all non-indexable, redirected, or API endpoints out of the sitemap.

## Redirect Rules For Sitemap Inclusion
Sitemap URLs should:
- never be redirected
- never be redirect sources
- always point to the final canonical version

Current note:
- `/services/` is a redirect route and correctly should stay out of the sitemap
- legacy service and city aliases should also stay out of the sitemap
